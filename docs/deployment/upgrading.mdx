---
title: "Upgrading"
description: "How to safely apply updates, run migrations, and restart the Stake Portal services."
---

Follow these steps in order every time you update to a new version of the portal. Skipping steps — especially running migrations after restarting the backend — can leave the application in an inconsistent state.

## Upgrade Steps

### 1. Pull the latest code

```bash
git pull origin main
```

### 2. Check the release notes

Before proceeding, review the changelog or release notes for the version you are upgrading to. Some releases include migration-specific instructions (e.g., data backfills, manual SQL steps, or configuration changes) that must be completed before or after running Alembic.

### 3. Apply database migrations

```bash
cd backend
uv run alembic -c alembic.ini upgrade head
```

<Warning>
  Always apply migrations **before** restarting the backend. Starting the backend against an unmigrated schema can cause runtime errors or data corruption.
</Warning>

This command is idempotent — it is safe to run even if no new migrations exist.

### 4. Rebuild the frontend

```bash
cd frontend
npm install
npm run build
```

`npm install` is required whenever `package.json` or `package-lock.json` has changed. Running it on every upgrade is safe and ensures dependencies are up to date.

### 5. Restart all services

**Docker Compose:**

```bash
docker compose down
docker compose up -d
```

Or, to rebuild images as well:

```bash
docker compose up -d --build
```

**Manual deployment:**

Restart each service process using whatever process manager you have configured (e.g., `systemctl restart stake-backend`, `pm2 restart stake-frontend`). Restart the backend first, then the frontend.

## Rolling Back a Migration

If you need to undo the most recently applied migration:

```bash
cd backend
uv run alembic -c alembic.ini downgrade -1
```

Each `-1` steps back one migration. Run `uv run alembic -c alembic.ini history --verbose` to see the full migration history and identify the target revision.

<Warning>
  Downgrading a migration may result in data loss if the migration added columns or tables that now contain data. Always back up the database before rolling back in a production environment.
</Warning>

## Checking the Current Migration State

```bash
cd backend
uv run alembic -c alembic.ini current
```

This prints the revision the database is currently at. Compare it against `uv run alembic -c alembic.ini heads` to verify the database is fully up to date.
